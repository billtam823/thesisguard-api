## Architecture

ThesisGuard is a stock investment thesis management API. The core workflow: a user adds a stock to their watchlist, generates an AI-powered buy thesis, saves news items, and triggers daily reviews where the AI assesses whether news materially changes the investment thesis. If it does, the system updates the stock's status and fires an alert.

### Package Layout

Each domain is fully self-contained (entity + repository + service + controller + DTOs):

- `stock/` — watchlist management. `Stock` is the root aggregate (ticker, exchange, status).
- `thesis/` — one `StockThesis` per stock (OneToOne). Generated by AI, editable by user.
- `news/` — `NewsItem` records (ManyToOne → Stock). Company news comes from **Seeking Alpha via RapidAPI** (`SeekingAlphaClient`, `GET /v1/symbols/news` on `seeking-alpha-finance.p.rapidapi.com`); it backs both ingest and the `/news/fetch` preview. SEC 8-K filings and Form 4 insider trades come from OpenBB's `sec` provider. The AI web-search collector (`AiNewsSearchClient`) is an optional supplement, **disabled by default** (`thesisguard.ai-news-search.enabled`). (newsfilter.io, Finnhub, and OpenBB-yfinance remain as unwired alternatives.) All sources merge in `NewsItemService.ingestLatest` and dedup via `NewsItem.dedupKey` (url+title hash). Live previews (no DB write): company news (`GET .../news/fetch`), filings (`GET .../news/filings`), insider (`GET .../news/insider`); SEC previews map into the same `FetchedNewsItemResponse` shape (source `SEC EDGAR`) and return empty for non-US-listed stocks (`Stock.isUsListed()`).
- `seekingalpha/` — `SeekingAlphaClient` wraps the Seeking Alpha RapidAPI (`GET /v1/symbols/news?category=all&ticker_slug=&page_number=`, header auth via `x-rapidapi-key`). The company-news source; returns empty when no `seekingalpha.api-key` is set so the rest of ingest still runs. Maps JSON:API `data[].attributes` (`title`, `publishOn`, `isPaywalled`, `content`) + `links.self` (→ `https://seekingalpha.com` + path) to `FetchedNewsItemResponse`.
- `newsfilter/` — `NewsfilterClient` wraps newsfilter.io's Query API (`POST /search`). Former news source, now unwired; kept as a fallback (re-wire `NewsItemService` + add `newsfilter.api-key` to use it).
- `finnhub/` — `FinnhubClient` wraps Finnhub's `/company-news`. Unwired fallback.
- `review/` — `DailyNewsReview` (ManyToOne → Stock) with child `NewsAnalysisItem` records per news item.
- `alert/` — `Alert` records triggered when a review elevates thesis change level.
- `ai/` — `AiClient` interface with two implementations (see below), plus `AiNewsSearchClient` (standalone, not part of `AiClient`): grounds an OpenRouter model with the `web_search` server tool to collect cited company news; no-ops to an empty list when `openrouter.api-key` is unset or `ai-news-search.enabled=false`.
- `openbb/` — `OpenBbClient` wraps the OpenBB Platform REST API.
- `common/exception/` — `ApiException` hierarchy + `GlobalExceptionHandler`.
- `config/` — CORS config (allows `localhost:5173` for the Vite frontend).

### Entity Relationships

```
Stock ──1:1──> StockThesis
     ──1:N──> NewsItem
     ──1:N──> DailyNewsReview ──1:N──> NewsAnalysisItem ──N:1──> NewsItem
     ──1:N──> Alert ──N:1──> DailyNewsReview (optional)
```

All relationships use `FetchType.LAZY`. Cascades (`ALL` + `orphanRemoval`) flow from parent to child.

### AI Client Strategy

`AiClient` interface has two beans with conditional activation:

- `OpenRouterAiClient` — active when `openrouter.api-key` is set (`@ConditionalOnProperty`). Calls `https://openrouter.ai/api/v1` chat completions with three models: `openrouter.model` for thesis generation, `openrouter.review-model` for the daily doctrine review, and `openrouter.triage-model` (cheap; defaults to the review model) for the news triage pre-filter (`AiClient.triageNews`). Prompts are loaded from `src/main/resources/prompts/` (incl. `news_triage_*`). Strips markdown fences from responses before JSON parsing.
- `MockAiClient` — active via `@ConditionalOnMissingBean(AiClient.class)` when no OpenRouter key. Keyword-based mock: news titles containing "broken/material/watch/minor" map to corresponding `ThesisChangeLevel` values; `triageNews` marks those keywords (plus fraud/recall/lawsuit/guidance cut) as material, the rest as noise.

### News Ingest vs Review (decoupled)

Ingestion and review are separate. `NewsItemService.ingestLatest`/`ingest` only fetch + dedup + store (news accumulates in the backlog with `reviewedAt = null`); they never call the review AI. `NewsFetchScheduler` (a `SchedulingConfigurer`, enabled via `@EnableScheduling`) runs ingest-only for every watchlist stock on each cron in `thesisguard.news-fetch.crons` (e.g. pre-market + lunch; empty list disables it; `thesisguard.news-fetch.zone` sets the timezone). Reviews stay user-triggered.

### Triage Gate

Before the expensive doctrine review, `DailyNewsReviewService` calls `AiClient.triageNews` to split the pending group into material vs noise (`review/NewsTriageResult`). The expensive `reviewNews` runs only on — and only reads — the material items; noise items are recorded as `NOISE` analysis rows at no AI cost. Triage fails safe (treat all as material on failure). Toggle with `thesisguard.triage.enabled` (default true).

### Stock Code Model

`Stock` carries three related fields:
- `ticker` — base symbol (e.g. `NVDA`, `RY`)
- `exchange` — friendly exchange name (e.g. `NASDAQ`, `TSX`); auto-populated from OpenBB equity profile on creation, or manually supplied
- `getProviderTicker()` — computed: US exchanges (NASDAQ, NYSE, AMEX, NYSE ARCA) return `ticker` as-is; others append `:EXCHANGE` (e.g. `RY:TSX`, `0700:HKEX`)

`StockResponse` exposes all three as `stock_code`, `exchange`, and `provider_ticker`.

### Path Variable Conventions

All stock-scoped endpoints use `{stockCode}` (not a numeric ID). `StockService.getEntity(String)` accepts either a numeric ID string or a ticker symbol and resolves both.

### Daily Review Workflow (`DailyNewsReviewService.runReview`, via `reviewPending`/`autoReview`)

1. Load stock entity + its saved thesis
2. Load pending (`reviewedAt == null`) `NewsItem` records for the stock — the accumulated group
3. If no pending news → save review with `No_News_Found`, skip AI
4. Triage the group (`AiClient.triageNews`) into material vs noise:
   - All noise → save a `No_Change` review (no `reviewNews` call, no memory update, no alert); mark all reviewed
   - Some material → call `AiClient.reviewNews()` on the **material items only** → save `DailyNewsReview` + `NewsAnalysisItem` records (noise items recorded as `NOISE`), update monitoring memory; mark all reviewed
5. Elevate `Stock.status` based on `ThesisChangeLevel`: `Watch_Change` → `Watch`, `Material_Change` → `Reduce_Review`, `Thesis_Broken` → `Sell_Review`
6. Create an `Alert` for `Watch_Change` and above

### Enum Serialization

`StockStatus` and `ThesisChangeLevel` use `@JsonValue`/`@JsonCreator` with human-readable labels (`"Hold"`, `"Watch Change"`, `"Thesis Broken"`, etc.) — not the enum constant names.

### Configuration

```yaml
# Key properties (application.yaml)
openbb.base-url: https://openbb.kingheung.com  # SEC filings/insider (sec provider); yfinance news is fallback-only
seekingalpha.host: seeking-alpha-finance.p.rapidapi.com  # company-news source (RapidAPI)
seekingalpha.api-key: <in gitignored application-local.yaml; RapidAPI key. Without it, news fetches return empty>
newsfilter.api-key: <unwired fallback news source>
finnhub.api-key: <unwired fallback news source>
openrouter.model: <model for thesis generation>
openrouter.review-model: <model for the daily doctrine review>
openrouter.triage-model: <cheap model for the news triage pre-filter; defaults to review-model>
openrouter.api-key: <in gitignored application-local.yaml at repo root; set to enable OpenRouterAiClient, omit to use MockAiClient>
thesisguard.triage.enabled: <true (default); false reviews every item>
thesisguard.news-fetch.crons: <list of cron exprs for automatic ingest-only fetch; empty disables>
thesisguard.news-fetch.zone: <ZoneId for the fetch crons, e.g. America/New_York>
```

Tests use `src/test/resources/application.yaml` which swaps the datasource to H2 (`MODE=PostgreSQL`) with `ddl-auto: create-drop`. No Docker needed for tests.

Schema is managed by Hibernate `ddl-auto: update` in production — no migration files.
